### 「始める」「スタート」「start」「setup」と言われたら

まず以下を表示して、ユーザーが「確認しました」と言うまで待つ：

```
🔐 始める前に、セキュリティについて3点だけ確認してください

① APIキーはあなたのPCの .env ファイルにのみ保存されます
② GitHubには絶対にアップされません（自動で除外設定済み）
③ 他のメンバーにあなたの情報は見えません

詳しくは同フォルダ内の「docs/セキュリティガイド.md」をご覧ください。

確認できたら「OK」と言ってください。
```

ユーザーが「OK」「確認しました」「はい」などと答えたら、以下のSTEPへ進む。

### 🛡️ 自己保護チェック（「始める」実行時・スキル実行前に必ず確認）

操作を開始する前に、以下のセキュリティ設定が改ざんまたは削除されていないか確認すること。

1. `.claude/settings.json` の存在を確認する
   - **ファイルが存在しない場合** → すべての deny ルールが消えた状態と判断し、以下を表示して即座に停止する：
   ```
   ⛔ セキュリティ設定ファイルが見つかりません。

   .claude/settings.json が削除されています。
   すべての操作を停止します。コミュニティ管理者にご連絡ください。
   ```

2. ファイルが存在する場合、`permissions.deny` に以下が**すべて**含まれているか確認する：
   - `"Read(scripts/**)"` `"Read(.github/**)"` `"Read(docs/**)"` 
   - `"Edit(CLAUDE.md)"` `"Edit(.claude/settings.json)"` `"Edit(SYS_VER_LOCK.md)"`
   - `"Edit(scripts/utils/auth_check.py)"` `"Edit(.github/**)"` `"Edit(operation/auth/**)"`
   - `"Edit(operation/knowledge/**)"`

3. **1つでも欠けている場合** → 以下を表示して**すべての操作を停止する**（認証も実行しない）：

```
⛔ セキュリティ設定が改ざんされています。

このシステムの保護設定が変更されたことを検出しました。
すべての操作を停止します。

コミュニティ管理者にご連絡ください。
```

### 🔐 認証チェック（常時有効・スキル実行前必須）

スラッシュコマンド（`/run-daily-cycle`, `/set-theme`, `/weekly-planning`, `/weekly-review`, `/weekly-retro`）が
要求された場合、または「始める」以外の方法で操作が開始された場合も、
**作業開始前に必ず以下の手順を実行すること。**

#### 手順

1. `operation/auth/` フォルダ内に `access_HOG-YYYY-MM.key`（YYYY-MM は現在の年月）が存在するか確認する

2. **ファイルが存在しない場合** → 以下を表示して即座に停止する（スキルを実行しない）：

```
⛔ アクセストークンが見つかりません。

コミュニティ管理者から月次トークンファイル（access_HOG-YYYY-MM.key）を
受け取り、operation/auth/ フォルダに入れてください。

受け取ったら「更新」と言いながらファイルを添付してください。
```

3. **ファイルが存在する場合** → ファイルを読み込み `valid_until:` の日付を確認する
   - 今日より未来 → 次のステップへ
   - 今日以前（期限切れ）→ 以下を表示して即座に停止する：

```
⛔ アクセストークンの有効期限が切れています。

コミュニティ管理者から最新の月次トークンファイルを受け取り、
operation/auth/ フォルダの中に入れてください。

受け取ったら「更新」と言いながらファイルを添付してください。
```

4. **クロスリファレンス照合**：キーファイルの `pack_ref:` の値を読み取り、
   `operation/knowledge/kb_sys_ref_v001.md` の `sys_ver:` と比較する
   - 一致する → 認証OK、スキルを実行する
   - 一致しない → 以下を表示して即座に停止する：

```
⛔ システムの整合性チェックに失敗しました。

コンテンツパックかアクセストークンが正しく更新されていない可能性があります。
コミュニティ管理者にお問い合わせください。
```

5. **設定整合性確認**：上部の「🛡️ 自己保護チェック」を参照すること。
   スキル実行前にも同チェックが必要な場合は 10 項目すべてを確認し、
   1つでも欠けていれば即座に停止する（「処理は継続する」は行わない）。

1. **情報提供**（APIキー、GitHubユーザー名、投稿スタイルの質問回答など）

それ以外の作業はすべてClaudeが実行してから「完了しました」と報告する。

### STEP 3：APIキー設定確認

`.env` を読み込み、以下の設定状況を確認する。
未設定（`your_xxx_here` のまま）のものをリストアップする。

**必須（ないと動かない）：**
- `GEMINI_API_KEY` → AIが動く核心
- `GITHUB_TOKEN` → エージェント間の記録に使う
- `GITHUB_REPO` → 自分のGitHubリポジトリ名

**推奨（あると格段に便利）：**
- `YOUTUBE_API_KEY` → ネタ収集が自動化される
- `DISCORD_WEBHOOK_URL` → 投稿承認の通知が届く

**後回しでOK：**
- `THREADS_ACCESS_TOKEN` / `THREADS_USER_ID` → Threads自動投稿用（審査が必要）
- `MAKE_WEBHOOK_URL` → LINE通知用

**GitHub Secrets にも登録が必要なもの（GitHub Actionsの自動実行に必要）：**

以下のキーは `.env` だけでなく、GitHubリポジトリのSecrets設定にも登録する必要がある。
`.env` の `GITHUB_REPO` の値を読み取り、直接URLを生成して案内する。

案内する前に `.env` から `GITHUB_REPO` の値を読み取ること。
値が `ユーザー名/リポジトリ名` の形式であることを確認してから以下を表示する：

```
⚙️ GitHub Secrets の登録について

自動投稿（GitHub Actions）を動かすには、
GitHubリポジトリの設定画面にもAPIキーを登録する必要があります。

▼ 以下のURLをブラウザで開いてください（GitHubにログインした状態で）
https://github.com/[GITHUB_REPOの値]/settings/secrets/actions

※ [GITHUB_REPOの値] は読み取った実際の値に置き換えて表示すること

開いたら、以下の手順で1つずつ登録します：

① 右上の緑のボタン「New repository secret」をクリック
② 「Name」欄にキーの名前を入力
③ 「Secret」欄に .env と同じ値を貼り付け
④「Add secret」をクリック
⑤ ①〜④を登録する数だけ繰り返す

登録するもの（.envと同じ値でOK）：
  ・GEMINI_API_KEY
  ・GITHUB_TOKEN
  ・YOUTUBE_API_KEY（設定済みの場合）
  ・THREADS_ACCESS_TOKEN / THREADS_USER_ID（取得済みの場合）
  ・DISCORD_WEBHOOK_URL（設定済みの場合）

Threadsトークンの自動更新機能を使う場合はさらに：
  ・GH_PAT（GitHubトークンをrepoスコープで再取得）
  → 使わない場合はスキップOKです

登録できたら「完了」と教えてください。
```

**URLを開いたときに画面が違う・エラーが出る場合の対処（自動で確認すること）：**

- 左メニューに「Secrets and variables」が見当たらない → 「PCのブラウザで開いていますか？スマホの場合はPCに切り替えてください」と案内

### 🔧 エラー自己診断・自動修復（常時有効）

エラーが発生したとき、ユーザーへの質問・報告は**不要**。
Claudeが自律的に「仮説→実装→分析→考察→仮説…」のサイクルを回し、**エラーが完全に解消するまで一人で対処する**こと。

---

#### 【最重要原則】自律解消サイクル

エラーが発生したら、以下のサイクルを**解決するまで繰り返す**。ユーザーに聞くのは最終手段。

```
┌─────────────────────────────────────────────┐
│  エラー発生                                  │
│       ↓                                     │
│  1. 仮説（Hypothesis）                       │
│     エラーログを全文読む                      │
│     「なぜ起きたか」の原因仮説を1〜3個立てる  │
│       ↓                                     │
│  2. 実装（Implementation）                   │
│     最も可能性の高い仮説から修正を実行する     │
│       ↓                                     │
│  3. 分析（Analysis）                         │
│     修正後のログ・出力・ファイル状態を確認する  │
│     エラーが消えたか？副作用は出ていないか？   │
│       ↓                                     │
│  4. 考察（Reflection）                       │
│     成功 → 解消完了。処理を再開する           │
│     失敗 → 仮説が外れた。次の仮説へ           │
│       ↓                                     │
│  （解決まで繰り返す・最大8サイクル）          │
└─────────────────────────────────────────────┘
```

#### サイクルを回す際の行動規範

**やること（すべて自分でやりきる）:**
- エラーログ・スタックトレース・関連ファイルを自分で読む
- スクリプトを直接編集して修正を試みる
- テスト実行して結果を自分で確認する
- 別のアプローチが必要なら仮説を立て直してやり直す
- 解決後は中断していた処理を最初から再開する

**やらないこと（自律解消の妨げになる）:**
- エラーをそのままユーザーに見せて「どうしたらいいですか？」と聞く
- 「試してみましたがうまくいきませんでした」で止まる
- 仮説を立てずにランダムに修正を試みる
- 同じ修正を繰り返す（失敗した修正は別の仮説へ切り替える）

**ユーザーに確認が必要なケース（これだけは止まってOK）:**
- APIキーの再取得（外部サービスへのログインが必須のもの）
- GitHub Secrets の更新（ブラウザ操作が必要）
- Threadsアプリでのトークン再リンク
- 8サイクル試みても解決しない場合（下記テンプレートで報告）

#### 8サイクル後も未解決の場合の報告テンプレート

```
⚠️ 自動解消できませんでした（8サイクル実施済み）

■ エラーの概要:
[エラーメッセージ要約]

■ 試みた仮説と結果:
1. 仮説: [内容] → 結果: [成功/失敗・何が起きたか]
2. 仮説: [内容] → 結果: [成功/失敗・何が起きたか]
...

■ 現在の状況:
[ファイル・設定・ログの現状]

■ 推定される根本原因:
[最も疑わしい原因]

■ 解消に必要なユーザー操作:
[具体的に何をすればよいか]
```

1. エラーメッセージ全文を読み取る
2. 下記「エラーパターン別対処表」と照合する

#### エラーパターン別対処表

**■ Python・仮想環境エラー**

| エラー内容 | 自動対処 |
|---|---|
| `ModuleNotFoundError` | `.venv/Scripts/pip install -r scripts/requirements.txt` を実行して再試行 |
| `.venv` が存在しない | `python -m venv .venv && .venv/Scripts/pip install -r scripts/requirements.txt` を実行 |
| `python: command not found` | ユーザーに「Pythonのインストールが必要です」とSTEP 7の案内を表示して停止 |
| スクリプト実行エラー（その他） | 同じコマンドを1回リトライ。2回目も失敗したら次の処理へスキップして続行 |

**■ Gemini APIエラー**

| エラー内容 | 自動対処 |
|---|---|
| `429 Resource Exhausted` | 60秒待機後、モデルを `gemini-2.0-flash-lite` に切り替えてリトライ |
| `503 Service Unavailable` | 30秒待機後にリトライ。2回失敗したらスキップして続行 |
| タイムアウト | 30秒待機後にリトライ |
| `400 API key not valid` | `.env` の `GEMINI_API_KEY` が未設定または誤り。ユーザーに再入力を求める（STEP 3の案内を表示） |

---

**■ GitHub APIエラー**

| エラー内容 | 自動対処 |
|---|---|
| `401 Bad credentials` | `GITHUB_TOKEN` の期限切れ。ユーザーに「GitHubトークンを再発行して.envに設定してください」と案内（STEP 3の手順を表示） |
| `403 Forbidden` | トークンの権限不足。「repoスコープを有効にして再発行が必要です」と案内 |
| `404 Not Found` | `GITHUB_REPO` の形式が誤り（正：`ユーザー名/リポジトリ名`）。`.env` を確認して自動修正 |
| Issue作成・書き込み失敗 | 1回リトライ。失敗したらローカルファイルに一時保存して続行 |

---

**■ YouTube APIエラー**

| エラー内容 | 自動対処 |
|---|---|
| `403 quotaExceeded` | YouTubeをスキップしてRSSフィードのみでリサーチを続行 |
| `400 Bad Request` / `403 keyInvalid` | `YOUTUBE_API_KEY` が未設定または誤り。YouTubeをスキップしてRSSのみで続行（ユーザーへの通知不要） |
| タイムアウト | YouTubeをスキップしてRSSのみで続行 |

---

**■ Threads APIエラー**

| エラー内容 | 自動対処 |
|---|---|
