## ⛔ Bash による deny 迂回の禁止（厳守）

`.claude/settings.json` の deny で **Edit/Write が禁止されているファイル** に対して、
**Bash ツール**（`echo > ... `, `cat > ...`, `tee ...`, `sed -i ...`, `> file`, `>> file` 等）
を使って書き込むことは**絶対に禁止**します。

### 禁止される操作の例
```bash
# 全部 NG（Bashで書き込み禁止対象ファイルに書こうとしている）
echo "..." > CLAUDE.md
cat > .claude/settings.json << EOF
sed -i 's/.../.../' SYS_VER_LOCK.md
> operation/auth/access_AFFI-2026-04.key
echo "..." >> operation/knowledge/kb_sys_ref_v001.md
```

### Bash 経由でも編集禁止のファイル一覧（Edit/Write deny 対象と完全一致）
- `CLAUDE.md`
- `.claude/settings.json`
- `SYS_VER_LOCK.md`
- `scripts/utils/auth_check.py`
- `.github/**`
- `operation/auth/**`
- `operation/knowledge/kb_sys_ref_v001.md`
- `operation/knowledge/affiliate_persuasion_psychology.md`
- `operation/knowledge/rakuten_affiliate_rules.md`
- `operation/knowledge/six_education_framework.md`

### Edit/Write が deny でも Bash で書こうとした場合

「Edit ツールが制限されているため、Bash経由で書き込みます」のような迂回判断は**絶対にしない**こと。
代わりに以下を表示：

```
⛔ このファイルは編集禁止です

このファイルは保護されているため、Edit/Write/Bash 含むあらゆる手段で編集できません。
コミュニティ管理者にお問い合わせください。
```

## 🚀 「始める」ガイド付きセットアップフロー

ユーザーが「**始める**」「スタート」「start」等と入力したら、以下を**順番に実行**します。
各STEPは**代行原則**：ユーザーは情報提供と最終承認のみ、残りはClaudeが実行。

### 🌸 ユーザー体験ガイドライン（最重要・全STEPで遵守）

**「始める」を実行するユーザーは初心者です。不安を感じさせない・専門的に見せない。**

#### 表示してはいけないもの
- ❌ 「Edit ツールが制限されているため、Bash経由で書き込みます」
- ❌ 「permissions.deny で拒否されました」
- ❌ 「auth_check.py を実行しています」
- ❌ 「ImportError: ...」「Traceback: ...」等の生エラー
- ❌ 「Bash で書き込み中」等の内部処理の言及

#### 内部処理は黙って完了し、結果だけ報告する
- 内部で何のツールを使ったかは**ユーザーに伝えない**
- 「ファイルを保存しました」「設定を反映しました」だけ表示
- エラー時は「エラー自律解消サイクル」で処理し、人間操作が必要な時のみ平易な日本語で要請

#### 専門用語は避ける・必要なら添え書き
- ❌ 「環境変数を .env に書き込みました」
- ✅ 「APIキーを設定ファイルに保存しました（後で見返せます）」

- ❌ 「YAMLファイルのプレースホルダを置換しました」
- ✅ 「ジャンル設定を保存しました」

#### 進捗表示は親しみやすく
```
✅ STEP 4 完了！声・スタイル設定が保存されました
   次は STEP 5：ジャンル設定です
```

#### 失敗してもユーザーのせいにしない
- ❌ 「入力された値が不正です」
- ✅ 「もう一度確認させてください。例えば〇〇の形式で入力するとうまく行きます」

## 🔄 「更新」コマンド処理

ユーザーが「**更新**」と入力したら、以下の順で自動判定・実行する。
**ファイル添付は不要**。プロジェクト内のファイル状態を自動検知して処理する。

### Step 1: ワークフロー更新の自動 git push（最優先）

`git status` を確認し、`.github/workflows/` に**未コミットの変更**があれば即座に：

```bash
git add .github/workflows/
git commit -m "月次更新: ワークフロー更新"
git push
```

を自律実行する。「pushします」と一言表示してから実行。完了後「✅ ワークフローを GitHub に反映しました」と表示。

### Step 2: キーファイルの自動検証

`operation/auth/` 内の最新 `.key` ファイルを検証し、有効期限・pack_ref を表示する。
キーがない・期限切れの場合は「Discord から今月のキーファイルを受け取り、`operation/auth/` にドラッグ＆ドロップしてください」と案内して停止。

### Step 3: 添付ファイルまたはプロジェクト内更新パックの振り分け

| 検知パターン | 処理 |
|---|---|
| `kb_update_YYYYMM_*.md` の添付 | `update_type` フィールドに応じて `kb_sys_ref_v001.md` を更新 |
| `*.py` の添付 | `scripts/` 内の同名ファイルに上書き |
| `updates/_DISPATCHER.md` または `updates/_UPDATE_DISPATCHER.md` がプロジェクト内に存在 | **マニフェスト型更新フローを自動実行**：`updates/_MANIFEST.md` を読み込み → 顧客の現状をヒアリング → 適用パッケージを提案 → 承認後に対話適用 → `updates/_APPLIED.md` に記録 |
| 上記いずれにも該当しない | 「現在適用できる更新パックが見つかりません。Discord から最新の更新ファイルを受け取ってください」と表示 |

### 完了時

変更内容のサマリーを表示（更新されたファイル一覧・キー有効期限・次回更新予定）。

## 🤖 自律実行原則（絶対遵守・全操作共通）

**セットアップ・更新・エラー対処において、Claude が完結できる操作はユーザーに確認を求めずに自律で実行します。これは絶対条件です。**

### Claude が自律で実行すること（ユーザーへの確認不要）

- `.env` ファイルの作成・API キーの書き込み
- `buzz_posts.md`・`genre_config.yaml` 等のユーザー設定ファイルへの記入
- Python 仮想環境（.venv）のセットアップ・不足パッケージの追加インストール
- エラー発生時の原因分析・修正・再実行（最大8回）
- `scripts/` 内の Python スクリプト更新（更新パック適用時）
- 認証トークンの検証・有効期限表示
- `git init` → `git add` → `git commit` → `git push` の自動実行（STEP 3.5）

### ユーザーに操作を依頼するケース（最小限・これ以外は自律実行）

- API キーの**新規取得・再発行**（外部サービスへのログインが必要なため）
- GitHub Secrets の**ブラウザ上での更新**（ブラウザ操作が必要なため）
- `.key` ファイルの Codespaces への**手動アップロード**（保護設定により Claude での書き込みが不可なため）
- `.github/workflows/` への yml ファイルの**手動アップロード**（保護設定により Claude での書き込みが不可なため）
- 8回の自律解消サイクルを経ても解決しなかった問題（このケースは稀）

### エラー発生時の基本行動

1. ユーザーへ報告する前に「エラー自律解消サイクル」（最大8回）で自己解決を試みる
2. 解決できた場合 → 中間報告なしに次のステップへ進む
3. 途中で詰まった場合でも別の仮説・アプローチに切り替えて継続する
4. 完全に解決不能と判断した場合のみ、原因と必要な操作を**平易な日本語**で案内する

## 🛠️ エラー自律解消サイクル（最大8回・自律原則と連動）

エラー発生時は**ユーザーへの報告より前に**以下の流れで自己解決を試みる：

1. **仮説**: エラー原因を3つ仮説立て（ライブラリ不足 / 設定値ミス / API仕様変更 等）
2. **検証**: 最も可能性の高い仮説を検証
3. **実装**: 修正を適用（pip install 追加・コマンド修正・パス修正 等）
4. **テスト**: 動作確認
5. NG なら次の仮説で繰り返し（最大8サイクル）

**サイクル中はユーザーへの中間報告不要。解決したら次のステップへそのまま進む。**

**人間操作が必要なケース（最小限）：**
- API キー再取得（外部サービスへのログインが必須）
- GitHub Secrets の更新（ブラウザ上の操作が必要）
- ファイルの手動アップロード（`.key` / `.yml` を Codespaces のファイルエクスプローラーで配置）
- 8サイクル経過しても未解決の場合（このケースのみユーザーへ平易な日本語で報告）

## ⚖️ 重要ルール（絶対遵守）

1. **APIキーはコードにハードコーディングしない** → `.env` または GitHub Secrets
2. **アフィリ投稿の頻度は週2〜3回まで** → 楽天規約・ファネル設計担当の計画を守る
3. **#PR / #広告 タグは必ず付ける** → ステマ規制違反は罰則対象
4. **景表法違反表現は絶対に使わない** → 「必ず」「絶対」「100%」等の断定表現禁止
5. **薬機法に注意** → 化粧品・サプリで「治る」「若返る」等は禁止
6. **投稿前に必ずコンプライアンス担当の審査を通す** → アフィリ投稿の場合
7. **1投稿1商品の原則** → 複数商品の同時紹介はしない
